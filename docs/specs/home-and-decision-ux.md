# ホーム画面と決定UX

このドキュメントは、ホーム画面と「決める」体験のUX方針を定める。
2026-06-30にデザイン検討を行い、本仕様の方向で進めることを決定した。
画面イメージのモックは [mockups/home-flow.html](../../mockups/home-flow.html)（全体フロー）と [mockups/home-B-refined.html](../../mockups/home-B-refined.html)（ホーム単体）にある。

関連する判断記録は [adr/0006-home-as-daily-pick.md](../adr/0006-home-as-daily-pick.md) を参照。

## 目的

アプリを開いた瞬間に「今日どこに行くか」を決められる状態にする。
ホームを保存一覧やマップにせず、開くだけで保存候補を思い出させ、最短で決定〜訪問〜記録へ導く。

## 対象外

- レストラン検索・発見（外部サービスに委ねる）
- 地図表示、経路検索、予約、営業時間の確認（決定後は外部マップに委ねる）
- 天気・現在地・時間帯など外部データを使った理由づけ（後述の通り採用しない）
- 複数人決定における招待リンク発行・アカウント・リアルタイム同期サーバー

## ユーザーストーリー

- アプリを開くと、保存候補の中からスコアリングで選ばれた「今日の一店」が1件だけ出る。気が向けばそのまま決定できる。
- 今日の一店が刺さらない日は、ガチャ・スワイプ・みんなで、のいずれかで決め直せる。
- 友人と一緒のときは、1台のスマホを回して全員の意思を反映して決められる。
- 決まったら外部マップで現地に向かう。
- 後日アプリを開くと、前回行った店を記録するかどうかをトップで促される。気が向かなければスキップできる。

## UXフロー

コアループは `保存 → 思い出す → 決める → 行く → 記録` （[user-flows.md](user-flows.md) 参照）。
ホームを起点に、以下の経路で構成する。

### ホーム（今日の一店）

- ヘッダーに日付と「今日の一店」、抽選母数（例: 「38店から抽選」）を表示する。
- スコアリングで抽選された1店をヒーローカードで提示する。
- カードに表示するのは**アプリ内にあるデータのみ**: 店名 / 行きたい度 / ジャンル / エリア・最寄駅 / 価格帯 / 保存からの経過日数 / 前回スキップからの経過日数 / タグ。
- 推論文（「今日は雨なので…」等）は出さない。抽選の理由は事実の提示（「保存から143日」「★5」）で代替する。
- 主アクションは「今日はここにする」、従アクションは「スキップ」。
- カード下部に「気分で選ぶ」として **ガチャ / スワイプ / みんなで** の3モードを常設する。
- 画面はスクロールなしで1画面に収める。

### 経路1: 今日の一店で即決 → 行く → 記録

1. 「今日はここにする」で決定画面へ。
2. 決定画面の主アクションは「Googleマップで開く」。営業時間・予約・経路は外部に委ねる。
3. 記録は後日通知ではなく、**次にアプリを開いたときトップに記録うながしを出す**（スキップ可）。
4. 記録画面では星評価を中心に、また行きたいか／写真／メモ／同行者を任意で追加できる。

### 経路2: ガチャ

1. 条件（エリア / ジャンル / 人数 / 予算 など）を選ぶ。選ぶほど母数が絞られ、件数を表示する。
2. 「この条件で回す」で1店を抽選。結果カードを表示する。
3. 「ここにする」で決定、「もう一回」で引き直し。

### 経路3: スワイプ

1. 候補カードを左右に振り分ける（候補にする / 今回は違う）。進捗を表示する。
2. 5件前後に絞り込んだら結果画面へ。
3. 結果画面では、**残った候補をタップして自分で1件を選ぶ**ことも、**ルーレット / ガチャでおまかせ決定**することもできる。手動とおまかせを並立させる。

### 経路4: みんなで（スマホ回し）

- 1台のスマホを友人間で回し、順番にタップする。招待リンク・アカウントは不要。
- 決め方は2バリエーションを用意する。
  - **投票制**: 1人1票。タップして次の人へ渡す。得票を累積表示する。
  - **順位制**: 候補に順位（＝スコア）をつける。全員の合計スコアが高い店に決定する。
- 全員の入力が終わると結果を表示し、経路1と同じ「行く → 記録」に合流する。

### 記録うながし（次回起動時）

- 直近に決定して未記録の訪問があるとき、ホーム上部にバナーを出す（例: 「昨日ことぶき行った？ 記録する / ✕」）。
- バナーは閉じればスキップできる。バナーの下には通常どおり今日の一店を表示する。

## 機能要件

- ホームは起動時に1日1店を抽選して提示する。抽選は保存済み・未アーカイブの店舗を母数とする。
- 抽選スコアは行きたい度・保存からの経過日数・前回提案/スキップからの経過日数などアプリ内データから算出する（詳細ロジックは別途）。
- スキップは `last_suggested_at` を更新し、当該店舗を再抽選プールに戻す。
- 決定はモードを問わず `DecisionSession` として記録し、決定店舗を保持する。
- 決定後は外部マップアプリを開くための手段を提供する。
- 未記録の決定が存在する場合、次回起動時にホームへ記録うながしを表示する。
- 記録は星評価のみで完了でき、その他項目は任意とする。

## データ要件

- 既存の `Restaurant`（`desire_level`, `created_at`, `last_suggested_at`, `last_visited_at`, `visited`, `archived`, タグ等）で今日の一店の抽選と表示要件を満たす。
- 決定モードは `DecisionSession.mode` に `gacha` / `swipe` / `vote` / `draft` / `tournament` / `roulette` を持つ。みんなでの投票制・順位制は `vote` を基本に表現する（順位制の扱いは未決、後述）。
- エリア・最寄駅・価格帯は保存時点の静的データを表示し、現在地からの実測距離は使わない。

## エッジケース

- 保存店舗が0〜1件しかない場合の今日の一店の出し方（抽選母数が成立しない）。
- 今日の一店をスキップし続けたときに同じ店が再浮上する頻度。
- 記録うながしを無視し続けたときのバナー表示回数（出しすぎと記録漏れのトレードオフ）。
- みんなで投票制で同票1位になった場合の決着。
- スワイプで候補が1件も残らなかった場合の導線。
- OGP画像が取得できない店舗のカード表示（写真なしフォールバックで対応。[mockups/home-B-refined.html](../../mockups/home-B-refined.html) 参照）。

## 未決事項

- 順位制のポイント配分（候補数が可変。上位3つに 3/2/1 を加点する案が有力だが未確定）。
- 記録うながしバナーの保持ポリシー（次回1回のみか、記録/明示スキップまで毎回か）。
- 投票制の同票決着方法（ルーレットへ流す等）。
- 今日の一店のスキップで即座に次の1店を出すか、当日は再表示しないか。
- 「徒歩6分」のような最寄駅からの所要表示を残すか、「渋谷駅近」のような表現に寄せるか。
- 抽選スコアの具体的な重み付け。
