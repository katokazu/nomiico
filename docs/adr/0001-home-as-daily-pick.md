# ADR: ホームをスコアリング抽選の「今日の一店」にする

## 状況

承認済み（2026-06-30）

## 背景

ホーム画面のデザインを検討した。複数の方向（決め打ちボールド、エディトリアル、プレイフル・ガチャ、ミニマル、再浮上スタック）を比較し、エディトリアル型のヒーローカード構成（旧B案）を基礎に選んだ。

その上で、当初の「今日のおすすめ」という枠組みには課題があった。おすすめという表現は、天気・現在地からの実測距離・時間帯といった外部データやAIによる理由づけを前提にしてしまい、MVPでの実現が難しく、プロダクトの焦点（決める体験）からもずれる。

検討すべき論点は次の通りだった。

- ホームの主役を何にするか（一覧・マップ・検索ではなく「決める」入口にしたい）。
- 1店を提示する根拠をどう説明するか（外部データに依存させたくない）。
- 各決定モードへの導線をどこに置くか。
- 決定後の「行く」「記録」をアプリ内でどこまで担うか。
- 複数人決定をどう成立させるか。

## 決定

- ホームは保存一覧・マップ・検索ではなく、**スコアリングで抽選された「今日の一店」を1日1件**提示する画面とする。
- 提示する1店の根拠は、**アプリ内にあるデータのみ**で表現する（行きたい度・保存からの経過日数・前回スキップからの経過日数・ジャンル・エリア・価格帯・タグ）。天気・現在地からの実測距離・時間帯などの外部データやAI推論文は使わない。
- ホームには「今日はここにする」「スキップ」に加え、**ガチャ / スワイプ / みんなで**の3モードを常設する。
- 「行く」は外部マップアプリに委ね、アプリ内で地図・予約・経路を完結させない。
- 記録は後日通知ではなく、**次回アプリ起動時にトップへ記録うながしを出し、スキップ可能**とする。
- 複数人決定は**1台のスマホを回す方式**とし、招待リンク・アカウントを不要とする。決め方は**投票制（1人1票）と順位制（順位＝スコアの合計）**の2バリエーションを用意する。
- スワイプの最終決定は、**手動で1件選ぶ導線と、ルーレット/ガチャによるおまかせ決定を並立**させる。

## 結果

容易になること。

- ホームの実装が外部API・AIに依存せず、保存済みデータだけで成立する。MVPで現実的。
- プロダクトの北極星（集めるより決める）にホームが直接寄与する。
- どの決定モードからも「行く → 記録」に合流するため、記録の蓄積が一本化される。
- 複数人決定がサーバー同期やアカウント連携なしで完結する。

難しくなること・制約。

- 「なぜこの店か」の説得力を外部文脈に頼れないため、抽選スコアの設計と事実提示の見せ方が重要になる。
- 外部マップに送り出すため、訪問検知を自動でできない。記録は次回起動時のうながしに依存する。
- 順位制と投票制の二系統を持つことで、決定セッションのデータモデルとUIに分岐が生じる。

## 検討した代替案

- **今日のおすすめ（AI・外部データ理由づけ）**: 説得力は高いが、外部依存とAIコストが大きく、MVPの焦点から外れるため不採用。
- **決め打ちボールド / プレイフル・ガチャ / ミニマル / 再浮上スタック**: いずれもホーム案として比較したが、1日1店の納得感と各モードへの導線を両立できるエディトリアル型を選択。
- **複数人決定の招待リンク・リアルタイム同期方式**: 実装コストが高く、対面でスマホを回すユースケースには過剰なため不採用。
