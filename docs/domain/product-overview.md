# プロジェクト概要

## 1. プロダクトコンセプト

このプロジェクトは、行きたい店をただ保存するためのアプリではなく、保存した候補の中から「今日どこに行くか」を最速で決めるためのアプリである。

> 保存するためではなく、行くためのアプリ。

Googleマップ、Instagram、食べログ、LINE、スクリーンショットなど、ユーザーはすでに多くの場所に「いつか行きたい店」を保存している。しかし、それらは散らばったまま見返されず、候補が増えすぎることで、実際には意思決定の役に立たなくなっている。

このアプリは、その散らばった「行きたい」を集約し、思い出させ、選びやすくし、実際の訪問と記録につなげる。

## 2. 解く課題

ユーザーの本当の課題は、店を見つけられないことではない。むしろ、すでに候補が多すぎることによって「今日どこに行くか」を決められないことにある。

代表的な課題は次の通り。

- Googleマップの保存ピンが増え続ける
- Instagramの保存投稿が埋もれる
- 食べログのお気に入りを見返さない
- LINEで送られた店のURLが流れていく
- 候補が多すぎて選べない
- 結局いつもの店に行ってしまう

このプロダクトの競合は、単なるグルメ検索サービスではなく、Googleマップの「保存済み」やInstagramの保存機能である。ただし、役割は競合というより補完に近い。外部サービスが「探す」「保存する」を担うのに対し、このアプリは「思い出す」「決める」「行く」を担う。

## 3. プロダクト思想

開発・機能追加の判断では、以下の原則を優先する。

1. 保存は3秒以内に終わる
2. アプリを開く理由は「決める」ときに絞る
3. 入力は必要になるまで求めない
4. 思い出す仕組みをプロダクトの中心に置く
5. 決める体験を楽しくする
6. 記録は思い出として自然に残る
7. 入力コストは未来の価値を超えてはいけない

通常の管理アプリのように、最初から詳細な情報入力を求めない。登録時は最低限の情報だけでよく、実際に決めるとき、行くとき、行ったあとに少しずつ情報が育つ設計にする。

## 4. コア体験

このアプリの基本体験は、次の流れで構成される。

```text
保存する
  ↓
思い出す
  ↓
決める
  ↓
行く
  ↓
記録する
```

### 保存する

Instagram、Googleマップ、食べログ、Webページ、LINEなどから共有されたURLを受け取り、できるだけ少ない操作で保存する。

MVPでは、以下が保存できれば成立する。

- URL
- 店名
- 保存日時
- 行きたい度
- ソース種別

ジャンル、エリア、タグ、価格帯などは自動取得できる範囲で補完し、必須入力にはしない。

### 思い出す

アプリを開かなくても、通知や提案によって保存済みの店を思い出せるようにする。

例:

- 「143日前に保存した焼鳥屋、まだ行きたい？」
- 「徒歩5分圏内に保存済みのお店があります」
- 「今週まだ行けていない人気候補があります」
- 「今日は暑いので、冷たい麺のお店はどうですか？」

通知は「整理してください」「入力してください」ではなく、行動につながるきっかけとして設計する。

### 決める

このアプリの主役は、保存一覧ではなく「決める」体験である。

主な決定モード:

- ガチャ: 条件を選ぶだけで1店舗を提案する
- スワイプ: 候補カードを左右に振り分けて絞る
- ドラフト: 複数人で順番に候補を選ぶ
- 投票: 各自が匿名で投票し、得票順に表示する
- トーナメント: 二択を繰り返して最終候補を決める
- ルーレット: 絞り込んだ候補から最後の1件をランダム決定する

ユーザーが本当に困るのは、候補を集めることではなく、最後の一歩を決めること。そのため、一覧や検索よりも、意思決定を助けるUIを優先する。

### 行く

決まった店はGoogleマップなど外部アプリで開き、実際の訪問につなげる。

この段階で重要なのは、アプリ内で完結させすぎないこと。地図、予約、経路検索などは既存サービスに委ね、このアプリは「決める」までの体験に集中する。

### 記録する

訪問後、自然に記録が残るようにする。最初は星評価だけでも十分とし、必要に応じて写真、メモ、同行者、再訪意向などを追加できる。

記録項目の例:

- 評価
- 誰と行ったか
- 写真
- メモ
- 食べたもの
- また行きたいか
- 訪問日時

## 5. MVPスコープ

最初のバージョンでは、機能を広げすぎず「保存した店から今日の候補を決められる」ことに集中する。

### 必須機能

- URL共有による店舗保存
- 手動での店舗登録
- 保存済み店舗一覧
- 店舗詳細
- 行きたい度の設定
- 訪問済み / 未訪問の管理
- 基本タグ
- ガチャモード
- スワイプモード
- 条件フィルタ
- 最終候補の決定

### あるとよい初期機能

- 保存から一定期間経過した店舗の通知
- 現在地付近の保存済み店舗通知
- 「まだ行きたい？」確認
- アーカイブ
- 訪問後の星評価

### 後回しにする機能

- 高度なAIコンシェルジュ
- 複雑な自動タグ付け
- 予約機能
- SNS機能
- 大規模なレコメンド基盤
- 完全なGoogleマップ / Instagramインポート（API直接連携）

ただし、完全な連携は後回しでも、**CSVによる軽量な一括取り込み**は「散らばった行きたいを集約する」という核心に直結するため、MVP直後の初期拡張として実施する（[specs/csv-import.md](../specs/csv-import.md)）。GoogleマップはTakeoutで保存リストをCSV化できる。

AI機能は将来的に有効だが、MVPでは「決める体験」の完成度を優先する。

## 6. 想定画面

### 保存完了画面

共有から保存した直後に、最低限の確認だけを行う。

```text
保存しました

〇〇食堂
行きたい度: ★★★★☆

[保存]
```

### ホーム / 起点画面

アプリを開いたら、迷わせず「今日どこ行く？」に入れる。

```text
今日どこ行く？

[ガチャで決める]
[みんなで決める]
[保存したお店]

今日のおすすめ
〇〇酒場
保存から143日
```

将来的には、ホーム画面すら薄くし、通知や共有リンクから目的の体験に直接入る設計も検討する。

### ガチャ画面

```text
条件

エリア
ジャンル
人数
シーン
予算

[決める]
```

結果カード:

```text
〇〇酒場
徒歩8分
保存から143日
行きたい度: ★★★★★

[ここにする]
[もう一回]
```

### スワイプ画面

```text
〇〇食堂
ジャンル: 焼鳥
予算: 3000円
タグ: 友達 / 駅近

[今回は違う] [候補にする]
```

5〜10件程度で候補を絞り、最後はガチャやルーレットで決める。

### 複数人で決める画面

初期拡張として、次のモードを検討する。

- ドラフト
- 匿名投票
- 全員スワイプ
- トーナメント
- ルーレット

複数人で使う場合は、遠慮や空気読みを減らし、本音で決めやすい体験を重視する。

## 8. 「今行くべき」スコア

将来的には、単なる行きたい度ではなく「今この店に行く理由」を提示する。

スコア要素の例:

- 保存から長期間経過している
- 現在地から近い
- 今日の天気や季節に合う
- 同行者やシーンに合う
- 行きたい度が高い
- 最近候補に出ていない
- 営業時間内である

提案例:

```text
今日のおすすめ: 〇〇酒場

徒歩8分です
保存してから143日経っています
友達との飲み会タグ付きです
今日は雨なので駅近です
```

この説明があることで、ユーザーは「なんとなくおすすめされた」ではなく「今日はここにしよう」と納得しやすくなる。

## 9. 将来構想

最初はレストランに特化する。ただし、根底にある価値は「保存されたまま消化されないものを、実際の行動に変える」ことである。

将来的には、次の領域にも拡張できる可能性がある。

- 行きたいカフェ
- 行きたい旅行先
- 読みたい本
- 見たい映画
- やりたいこと
- 買いたいもの

このアプリは、広い意味では「行きたい場所・やりたいことを、実際の体験へ変換する意思決定エンジン」になり得る。

ただし、初期段階では拡張性よりも尖りを優先し、「保存した飲食店から今日行く店を決める」体験に集中する。

## 10. 判断基準

新しい機能を追加する際は、次の問いに答えられるかを確認する。

- 保存が面倒にならないか
- 決める体験が速くなるか
- 候補が埋もれにくくなるか
- ユーザーに不要な入力を求めていないか
- 実際の訪問につながるか
- 記録が自然に残るか
- Googleマップや食べログの単なる劣化版になっていないか

このプロジェクトの中心は、店舗情報の網羅性ではなく、ユーザー自身がすでに持っている「行きたい」を実際の行動に変えることにある。
